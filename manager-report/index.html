Executive summary

⚠

The problem

Users build strategies in StrykeX but backtests time out. QuestDB is queried too heavily, indicators are recomputed every click, and Python loops are too slow.

⚡

The solution

ODIN adds Redis + Parquet caches, precomputes indicators nightly, and runs a Numba-accelerated engine. StrykeX calls ODIN instead of doing everything inline.

✓

What we need

Approval to wire production backtest API to ODIN, provision Redis + ETL, and connect QuestDB with bounded queries. Prototype is already built and benchmarked.

Problem statement

When a user completes the StrykeX builder flow and clicks Backtest:

Step 1Strategy OverviewPick NIFTY, segment, intraday/positional
Step 2Strategy Setup5 Minute chart, spot/futures
Step 3Entry & ExitRSI, EMA, conditions
Step 4BacktestClick → timeout error

What breaks in production

✕Backend fetches too much OHLC — often full 5-year history
✕Every indicator recomputed on every run, all candles
✕Single-threaded Pandas/Python execution
✕HTTP times out — “Oops timeout error”
ODIN sits between StrykeX and QuestDB. The Strategy Builder UI stays unchanged. Only the backtest API backend is redirected or augmented.

04

Target architecture

Three data tiers — most requests never touch QuestDB

L1RedisHot cache — OHLC & indicator slices, 24h TTL, sub-ms reads on repeat backtests
L2ParquetMonthly columnar files on SSD — 10–30× faster than SQL for range scans
L3QuestDBSource of truth — bounded queries only, partitioned by month/day
Why timeouts stop: OHLC + common indicators served from Redis or Parquet in milliseconds. Heavy 5-year compute moves to nightly batch at 11 PM IST, not per user click.

06

QuestDB query contract

Phase 0 quick win — every query must be bounded. API rejects unbounded date ranges.

SELECT timestamp, open, high, low, close, volume
FROM ohlc_5m
WHERE symbol = 'NIFTY'
  AND timestamp >= '2024-01-01T00:00:00.000000Z'
  AND timestamp <  '2025-01-01T00:00:00.000000Z'

DDL requirements

  • Partition by MONTH or DAY on timestamp
  • Filter symbol + timeframe in SQL — not in Python
  • Never SELECT *
  • Timestamp column aligned with StrykeX 5m candles
07

Parquet layout

data/parquet/
  ohlc/NIFTY/5m/          →  2024-01.parquet, 2024-02.parquet, ...
  indicators/NIFTY/5m/    →  rsi_14, ema_9, ema_20, atr_14, bb_*, macd_*
  catalog.json            →  { min_ts, max_ts, months[] }

Nightly ETL exports NIFTY 5m only initially. All other symbols/timeframes remain in QuestDB until Phase 6.

08

Indicator strategy

Tier A — Nightly precompute

EMA, SMA, RSI, MACD, ATR, Bollinger — standard params used ~80% of the time. Stored in Parquet.

Tier B — On-demand + cache

Custom params (e.g. RSI 21) computed once with Polars, cached in Redis as ind:NIFTY:5m:RSI:21:start:end. 96 StrykeX indicators catalogued with entry/exit rule templates in PostgreSQL.

Phased rollout

Phase 0 — Baseline & QuestDB fixes

Week 1Done

Goal: Measure the timeout; quick wins without new infra.

  • Profile NIFTY 5m backtest — QuestDB, indicators, loop, serialization
  • Record p50/p95 vs gateway timeout (30–60s)
  • Apply bounded QuestDB query contract
  • Document StrykeX API schema
Exit: Baseline JSON; 1-year NIFTY query under 2s

Phase 1 — Parquet tier

Week 2Done

Goal: 10–30× faster analytical reads.

  • Nightly QuestDB → monthly Parquet export
  • Polars scan_parquet read path
  • QuestDB only on cache miss
Exit: 1-year OHLC load under 500ms from Parquet

Phase 2 — Redis cache

Week 3Done

Goal: Fix “same strategy, second run” timeouts.

  • Keys: ohlc:NIFTY:5m:{start}:{end} and indicator keys
  • 24h TTL, invalidate after nightly ETL
  • /metrics endpoint for cache hit rate
Exit: Repeat backtest under 500ms total

Phase 3 — Indicator precompute

Week 4Done

Goal: Kill redundant indicator compute on every click.

  • Tier A → Parquet; Tier B → on-demand + Redis
  • Align with StrykeX UI indicator names
Exit: 3+ indicators: under 5s cold / 1s warm

Phase 4 — Fast execution engine

Week 5Done

Goal: 10×+ faster condition evaluation.

  • Polars prep → Numba JIT candle loop
  • Map Equal, Greater Than, Crosses Above, OR case blocks
Exit: Condition loop 10× faster than Python

Phase 5 — StrykeX integration

Week 6Done

Goal: Users stop seeing timeout errors.

  • USE_ODIN=true feature flag on StrykeX API
  • Drop-in adapter: POST /v1/backtest/strykex
  • Do not rely on increasing HTTP timeout as the fix
Exit: NIFTY 5m backtest completes in staging UI

Phase 6 — Scale beyond NIFTY 5m

Week 7+Done

Goal: Full QuestDB catalog.

  • BANKNIFTY + remaining instruments
  • 1m, 15m, 1h timeframes
  • Ray for multi-instrument grid sweeps
