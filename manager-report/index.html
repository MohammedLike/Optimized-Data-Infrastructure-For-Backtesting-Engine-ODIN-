In one sentence: We built a speed layer behind StrykeX so backtests finish in under a second instead of timing out after 60+ seconds.

😤 The problem users face today

When a trader builds a strategy on StrykeX and clicks Backtest, they often get an error — the system takes too long and gives up.

Why it breaks

  • Too much data — the system pulls years of price history every single click
  • Repeats the same work — indicators like RSI are calculated from scratch every time
  • Slow processing — one request does everything in a single slow path
  • User gives up — after ~60 seconds, the page shows a timeout

✅ The fix — Project ODIN

ODIN (Optimized Data Infrastructure) sits behind StrykeX. Users see the same website. Only the back-end gets smarter and faster.

No UI changes. Traders keep using Strategy Builder exactly as today. We only change where the backtest request goes.

Think of it like a restaurant

❌ Today Every order sends the waiter to the farm, harvests crops, cooks from scratch — then the customer waits an hour.
→
✅ With ODIN Popular dishes are prepped overnight. The kitchen grabs ready ingredients. Orders come out in minutes.

What ODIN does (plain English)

PieceWhat it does
Price databaseStores all NIFTY candle data (QuestDB) — the master copy
Fast copiesKeeps ready-to-use monthly files (Parquet) — like pre-sliced ingredients
Memory cacheRemembers the last request (Redis) — instant if user runs same test again
Pre-calculated indicatorsRSI, EMA, MACD etc. computed overnight, not on every click
Fast engineRuns the strategy rules on optimized code instead of slow Python loops

What happens when user clicks Backtest
sequenceDiagram
    participant U as Trader
    participant S as StrykeX
    participant O as ODIN
    U->>S: Click Backtest
    S->>O: Send strategy
    O->>O: Get prices from cache or ready files
    O->>O: Load pre-calculated indicators
    O->>O: Run strategy rules
    O-->>S: Trades and P&L
    S-->>U: Show results quickly
Key idea: Heavy work moves to overnight batch jobs (11 PM). User clicks only trigger light, fast lookups — not full re-processing.

🙋 What we need from you

To move from prototype to production, we need:

  • Approval to connect the live StrykeX backtest API to ODIN (no UI change)
  • Access to production price database (QuestDB) credentials
  • Confirmation of default backtest date range (e.g. 1 year vs 5 years)
  • Staging test — one week to validate with real users before full rollout
What you do NOT need to worry about: New screens, user training, or product redesign. This is invisible infrastructure — users just stop seeing timeout errors.
